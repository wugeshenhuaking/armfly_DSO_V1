# 基于uCOS-III+STemWin的数字示波器项目分析

## 项目概述

本项目是基于 **STM32F4** 的数字示波器（DSO），核心技术栈：

| 模块 | 技术 |
|------|------|
| MCU | STM32F4xx (Cortex-M4) |
| 操作系统 | uCOS-III |
| GUI框架 | STemWin V5.28 (Segger emWin) |
| 文件系统 | FatFS R0.11 |
| 信号处理 | ARM DSP Lib (CMSIS-DSP) |
| LCD接口 | FSMC (8080并口) |
| 外部SRAM | IS61WV102416BLL (2MB, 16Bit, 10ns) |
| LCD控制器 | RA8875 / ILI9488 / SPFD5420 (多屏兼容) |
| 分辨率 | 800x480 (主要支持7寸屏) |

## 一、FSMC 如何同时驱动 LCD 屏和外置 SRAM

### 1.1 FSMC Bank 分配策略

STM32F4 的 FSMC Bank1 包含 4 个 NOR/SRAM 子区域（NE1~NE4），本项目通过**不同的 Bank 片选**来分别驱动 LCD 和 SRAM：

| 设备 | FSMC Bank | 片选引脚 | 基地址 | 数据位宽 |
|------|-----------|----------|--------|----------|
| **LCD 屏幕** | `FSMC_Bank1_NORSRAM4` | PG12 (NE4) | `0x6C000000` | 16-bit |
| **外部 SRAM** | `FSMC_Bank1_NORSRAM3` | PG10 (NE3) | `0x68000000` | 16-bit |

> 注：FSMC Bank1 基地址从 `0x60000000` 起，每个 Bank 占用 `0x04000000`（64MB）地址空间。Bank3 = `0x68000000`，Bank4 = `0x6C000000`。

### 1.2 外部 SRAM 驱动配置

文件：`User/bsp_stm32f4xx/src/bsp_fsmc_sram.c`

```c
// SRAM 地址和大小定义
#define EXT_SRAM_ADDR   ((uint32_t)0x68000000)
#define EXT_SRAM_SIZE   (2 * 1024 * 1024)  // 2MB
```

FSMC Bank3 配置要点（`bsp_InitExtSRAM` 函数）：

- **GPIO 分配**：使用 PD0~PD15、PE0~PE15、PF0~PF5/PF12~PF15、PG0~PG5 地址线和数据线
- **片选信号**：PG10 (`FSMC_NE3`)
- **时序配置**：
  - `FSMC_AddressSetupTime = 3`（地址建立时间 3 个 HCLK）
  - `FSMC_DataSetupTime = 2`（数据建立时间 2 个 HCLK）
  - `FSMC_AccessMode = FSMC_AccessMode_A`（模式A，无地址复用）
- **存储器类型**：`FSMC_MemoryType_SRAM`
- **数据位宽**：16-bit
- **地址线**：使用 PF0~PF5 + PG0~PG5 共 12 根地址线（A0~A11），加上 PE3~PE5 参与译码（A19~A21），总共支持 1MB 寻址空间（实际 SRAM 容量 2MB，使用 16-bit 数据宽度时地址空间 1M 字 = 2MB）

### 1.3 LCD 屏驱动配置

文件：`User/bsp_stm32f4xx/src/bsp_tft_lcd.c`

FSMC Bank4 配置要点（`LCD_FSMCConfig` 函数）：

- **GPIO 分配**：数据线 D0~D15 共 16 根，复用 PD/PE 引脚；读信号 NOE (PD4)，写信号 NWE (PD5)
- **片选信号**：PG12 (`FSMC_NE4`)
- **RS 信号**：PD13 (`FSMC_A18`)，用于区分命令/数据（地址 0 写命令，地址 1 写数据）
- **读写时序分离配置**：
  ```c
  // 写时序
  timingWrite.FSMC_AddressSetupTime = 4;
  timingWrite.FSMC_DataSetupTime = 6;
  
  // 读时序
  timingRead.FSMC_AddressSetupTime = 4;
  timingRead.FSMC_DataSetupTime = 8;
  ```
- **存储器类型**：`FSMC_MemoryType_SRAM`（将 LCD 控制器当做 SRAM 设备访问）
- **数据位宽**：16-bit

### 1.4 LCD 底层访问方式

LCD 的命令/数据访问通过 FSMC 地址映射实现：

```c
// 典型的 8080 接口 LCD 操作方式
// 基地址 + 0x00000000 → 写命令（RS=0）
// 基地址 + 0x00000002 → 写数据（RS=1，A18=1）
// 基地址 + 0x00000000 → 读数据（NOE 控制）
```

由于使用 FSMC Bank4 (NE4) 且 RS 接在 A18 上：
- **写命令**：向 `0x6C000000` 写入 16-bit 数据
- **写数据**：向 `0x6C00040000` 写入 16-bit 数据（A18=1 → 地址偏移 `1<<18 = 0x40000`）

具体像素操作封装在 LCD 控制器驱动中（如 `RA8875_PutPixel`、`ILI9488_PutPixel`）。

### 1.5 地址线复用

LCD 和 SRAM **共享数据总线 D0~D15**（PD0~PD1, PD8~PD15, PE7~PE15），以及读信号 NOE (PD4)、写信号 NWE (PD5)。区别在于：

- **片选不同**：LCD 用 NE4 (PG12)，SRAM 用 NE3 (PG10)
- **地址线部分共享**：部分地址线（PD11~PD13, PE4~PE5）在两设备中使用
- **SRAM 专用地址线**：PF0~PF5, PG0~PG5 等

FSMC 通过片选信号自动仲裁总线，无需软件干预。

## 二、emWin 示波器如何绘制

### 2.1 整体架构

emWin 的显示驱动采用**自定义 Template 驱动**（`GUIDRV_Template`），通过将底层 LCD 操作函数桥接到 emWin 驱动接口。

核心文件关系：

```
LCDConf.c          → LCD_X_Config() 配置显示驱动
GUIConf.c          → GUI_X_Config() 分配 emWin 内存（外部SRAM）
GUIDRV_Template.c  → 自定义显示驱动，桥接 emWin 与 LCD 硬件
bsp_tft_lcd.c      → LCD 底层硬件驱动（RA8875/ILI9488/SPFD5420）
LCD_RA8875.c       → RA8875 LCD 控制器驱动
LCD_ILI9488.c      → ILI9488 LCD 控制器驱动
```

### 2.2 emWin 驱动配置

文件：`emWin/Config/LCDConf.c`


### 2.3 自定义显示驱动 (GUIDRV_Template)

文件：`emWin/DisplayDriver/GUIDRV_Template.c`

这是连接 emWin 和硬件 LCD 的**核心桥接层**。关键函数：

| 函数 | 功能 | 硬件操作 |
|------|------|----------|
| `_SetPixelIndex` | 设置单个像素 | `RA8875_PutPixelGUI()` 或 `ILI9488_PutPixel()` |
| `_GetPixelIndex` | 读取单个像素 | `RA8875_GetPixelGUI()` 或 `ILI9488_GetPixel()` |
| `_DrawHLine` | 绘制水平线 | `RA8875_DrawHLine()` 或 `ILI9488_DrawHLine()` |
| `_DrawVLine` | 绘制垂直线 | `RA8875_DrawVLine()` 或 `ILI9488_DrawVLine()` |
| `_FillRect` | 填充矩形 | `RA8875_RTERect()` 或 `ILI9488_FillRect()` |
| `_DrawBitLine16BPP` | 绘制16bpp位图行 | `RA8875_DrawHColorLine()` 或 `ILI9488_DrawHColorLine()` |

所有函数通过全局变量 `g_ChipID` 判断当前 LCD 芯片类型，分派到对应驱动。

优化策略（`emWin_Optimize = 1`）：
- 水平线、垂直线、填充矩形、位图行绘制均调用 LCD 控制器的**硬件加速**函数
- 例如 `RA8875_DrawHLine` 利用 RA8875 内置的 BTE（Block Transfer Engine）进行快速绘制

#### 2.4.3 波形绘制核心函数 `_Draw()`

文件：`emWin/emWinTask/MainTask.c` (第 552~684 行)

```c
static void _Draw(void * p) 
{
    PARAM * pParam = (PARAM *)p;

    // 第1步：绘制示波器网格背景
    DSO_DrawBakFrame();

    // 第2步：ADC 数据 → 屏幕坐标转换
    for(i = 0; i < 600; i++)
    {
        temp = pParam->aY[i] * 100 / g_AttTable[Ch1AmpId][0];
        temp = temp + g_usRefPos - 40;
        pParam->aY[i] = 400 - temp + (g_usRefPos - 240) * 2;
        aPointsRe[i].y = pParam->aY[i];
    }

    // 第3步：绘制波形曲线（支持实线/虚线/点线/点划线）
    GUI_SetColor(GUI_YELLOW);
    GUI_SetLineStyle(g_ucLineStyle);
    GUI_DrawPolyLine(aPointsRe, 600, DSOSCREEN_STARTX, DSOSCREEN_STARTY);

    // 第4步：绘制 FFT 幅频曲线（红色，可选）
    if (FFT条件满足) {
        GUI_SetColor(GUI_RED);
        GUI_DrawPolyLine(&aPointsRe[2], 598, ...);
    }

    // 第5步：绘制触发游标
    GUI_FillPolygon(&aPointsTrig[0], ..., 628, g_usTriPos);

    // 第6步：绘制测量游标（水平/垂直，可选）
    if (hWinCoursorFlag == 1) DSO_DrawCursorH();
    if (hWinCoursorFlag == 2) DSO_DrawCursorV();
}
```

### 2.5 ADC 数据采集

- 使用 **3 个 ADC** (ADC1, ADC2, ADC3) + **DMA** 循环采集
- 每个 ADC 有 10K 的 16-bit 缓冲区，共 10K x 3 = 30K 采样点
- 单通道最高采样率：2.8Msps
- 触发方式：自动触发（软件上升沿检测）和普通触发（ADC 模拟看门狗）

## 三、显示缓存区使用分析

### 3.1 emWin 内存分配

文件：`emWin/Config/GUIConf.c`

```c
#define EX_SRAM 1  // 使用外部 SRAM

#if EX_SRAM
    #define GUI_NUMBYTES (1024 * 1024 * 2)  // 分配 2MB 外部 SRAM 给 emWin
#else
    #define GUI_NUMBYTES (70 * 1024)        // 仅 70KB 内部 SRAM
#endif
```

```c
void GUI_X_Config(void)
{
    static U32 *aMemory;
    aMemory = (U32 *)EXT_SRAM_ADDR;  // 直接使用 0x68000000 外部 SRAM
    GUI_ALLOC_AssignMemory(aMemory, GUI_NUMBYTES);  // 2MB
    GUI_ALLOC_SetAvBlockSize(GUI_BLOCKSIZE);        // 最小块 0x80 = 128 字节
}
```

**结论**：emWin 的所有动态内存分配（窗口、控件、字体缓存、绘图上下文等）都从**外部 SRAM** 中分配，使用了全部 2MB 外部 SRAM 空间。

### 3.2 emWin Memory Device（显存设备）

**关键发现**：本项目**使用了 emWin 的 Memory Device（显存设备）机制**来绘制波形。

文件：`emWin/Config/GUIConf.h`
```c
#define GUI_SUPPORT_MEMDEV   1    // 使能 Memory Device 支持
```

文件：`emWin/emWinTask/MainTask.c` 第 1024 行
```c
GUI_MEMDEV_Draw(&Rect, _Draw, &Param, 0, GUI_MEMDEV_NOTRANS);
```

**`GUI_MEMDEV_Draw` 的含义**：

这个函数是 emWin 的**分片存储绘制**（Tiled Rendering）机制，工作流程如下：

1. emWin 在内部创建一个与 `Rect` 区域（600x400 像素）同等大小的 Memory Device
2. Memory Device 的像素数据存储在 emWin 的内存池中（即外部 SRAM）
3. 调用 `_Draw()` 回调函数，所有绘制操作先写入 Memory Device（而不是直接写 LCD）
4. 绘制完成后，将 Memory Device 的内容一次性传输到 LCD

**好处**：
- **消除闪烁**：波形刷新时不会出现"先清屏再画线"导致的闪烁
- **提高效率**：减少 FSMC 总线切换次数（数据在 SRAM 内部组装好后一次性输出）
- **支持叠加**：网格背景和波形曲线在 Memory Device 中合成后一次性显示

600x400 像素 x 16-bit = 480,000 字节 ≈ **469KB** 的显存占用（每次刷新波形时使用）。

### 3.3 波形数据缓冲区

```c
// 6KB 波形缓冲区（3通道 x 2K 采样点）
uint16_t g_usWaveBuf[1024*6];     // 主显示缓冲区
uint16_t g_usWaveBuf1[1024*6];    // 临时缓冲区（从 DMA 复制）

// 600 个点的屏幕坐标缓存
GUI_POINT aPointsRe[600];

// FFT 运算缓冲区
float32_t testInput_fft_2048[2048];   // 2048 点 FFT 输入
float32_t testOutput_fft_2048[2048];  // 2048 点 FFT 输出
```

### 3.4 RA8875 内部显存

当使用 RA8875 LCD 控制器时，RA8875 芯片内部自带显存（VRAM）：

- 800x480x16-bit = 768KB 内部 VRAM
- emWin 可以通过 `LCD_X_SETVRAMADDR` 指令配置 VRAM 地址
- RA8875 支持硬件加速操作（BTE 引擎），支持矩形填充、位图传输、水平线绘制等

### 3.5 缓存区总结

| 缓存区 | 位置 | 大小 | 用途 |
|--------|------|------|------|
| 外部 SRAM | 0x68000000 | 2MB | emWin 全局内存池（含控件、字体、Memory Device） |
| Memory Device | 外部 SRAM (emWin 池内) | ~469KB | 波形绘制离屏缓冲（600x400x16bit） |
| RA8875 内部 VRAM | LCD 控制器 | 768KB | LCD 帧缓冲（RA8875 方案） |
| g_usWaveBuf | 内部 SRAM | 12KB | ADC 波形数据缓冲 |
| g_usWaveBuf1 | 内部 SRAM | 12KB | ADC 数据临时缓冲 |
| testInput/Output | 内部 SRAM | 16KB | FFT 运算缓冲 |
| aPointsRe | 内部 SRAM | ~7KB | 屏幕坐标点数组 |

## 五、是否开启了全屏显示缓冲区（Framebuffer）？

### 5.1 结论

**本项目没有开启全屏帧缓冲区（Framebuffer），但有局部离屏缓冲机制。**

具体来说：

| 缓冲方案 | 是否使用 | 说明 |
|----------|----------|------|
| **全屏 MCU 侧 Framebuffer（800x480）** | **否** | 不存在一个完整的帧缓冲区在 MCU 内存中先渲染再传输 |
| **emWin Memory Device 局部缓冲（600x400）** | **是** | 波形绘制区域使用了离屏缓冲，渲染完再拷贝到 LCD |
| **RA8875 内部 VRAM（768KB）** | **是（仅 RA8875）** | LCD 控制器芯片内部自带帧缓冲 |
| **LCD_X_SHOWBUFFER 多缓冲** | **否** | 未实现多缓冲切换机制 |

### 5.2 详细分析

#### 5.2.1 全屏 Framebuffer 未启用的证据

**证据1：`VRAM_ADDR` 为 0，无显存地址配置**

`emWin/Config/LCDConf.c` 第 97~99 行：
```c
#ifndef VRAM_ADDR
#define VRAM_ADDR 0    // 没有指定显存地址，0 表示无效
#endif
```

**证据2：`LCD_X_SETVRAMADDR` 是空实现**

`emWin/Config/LCDConf.c` 第 173~180 行：
```c
case LCD_X_SETVRAMADDR:
    // 注释写着 "Required for setting the address of the video RAM"
    // 但实际什么都没做，直接 return 0
    return 0;
```

如果项目使用了全屏 Framebuffer，这里应该会把 VRAM 地址配置到 LCD 控制器。

**证据3：`LCD_X_SHOWBUFFER` 是空实现**

`emWin/Config/LCDConf.c` 第 191~197 行：
```c
case LCD_X_SHOWBUFFER:
    // 注释写着 "Required if multiple buffers are used"
    // 但实际什么都没做
    return 0;
```

emWin 的多缓冲机制（双缓冲/三缓冲）需要实现 `LCD_X_SHOWBUFFER` 来切换缓冲区。本项目没有实现，说明**没有使用 emWin 的多缓冲功能**。

**证据4：显示驱动直接操作 LCD 硬件**

`emWin/DisplayDriver/GUIDRV_Template.c` 中，所有绘制函数都是直接调用 LCD 控制器函数：

```c
// _SetPixelIndex → 直接写 LCD 控制器
RA8875_PutPixelGUI(xPhys, yPhys, PixelIndex);  // 写 RA8875
ILI9488_PutPixel(xPhys, yPhys, PixelIndex);     // 写 ILI9488

// _FillRect → 直接操作 LCD 硬件
RA8875_RTERect(x0, y0, x1, y1, ColorIndex);    // RA8875 硬件加速填充
ILI9488_FillRect(x0, y0, x1 - x0 + 1, ...);     // ILI9488 直接填充
```

没有任何中间缓冲层。每次 `GUI_DrawPoint()`、`GUI_DrawHLine()` 等调用都会通过 FSMC 总线直接写 LCD 控制器。

#### 5.2.2 波形区域的局部离屏缓冲（Memory Device）

虽然没有全屏 Framebuffer，但项目**对波形绘制区域（600x400 像素）使用了 emWin 的 Memory Device 机制**，实现了"先渲染到内存，再传输到 LCD"的效果。

**配置使能**：`emWin/Config/GUIConf.h` 第 68 行：
```c
#define GUI_SUPPORT_MEMDEV   1    // 使能 Memory Device 支持
```

**驱动支持**：`emWin/DisplayDriver/GUIDRV_Template.c` 第 852~858 行：
```c
#if GUI_SUPPORT_MEMDEV
case LCD_DEVDATA_MEMDEV:
    return (void *)&GUI_MEMDEV_DEVICE_16;  // 返回 16bpp Memory Device 描述符
#endif
```

**使用方式**：`emWin/emWinTask/MainTask.c` 第 1022~1024 行：
```c
/* 采用分片存储空间绘制 */
GUI_MEMDEV_Draw(&Rect, _Draw, &Param, 0, GUI_MEMDEV_NOTRANS);
```

**工作流程**：

```
Step 1: emWin 在外部 SRAM 中分配一个 600x400x16bit ≈ 469KB 的 Memory Device
Step 2: 调用 _Draw() 回调函数，所有绘制操作（网格+波形+游标）写入 Memory Device
        ↓ （此时 LCD 上没有任何变化）
Step 3: 绘制完成后，emWin 将 Memory Device 的内容逐行拷贝到 LCD
        → _DrawBitmap → _DrawBitLine16BPP → RA8875_DrawHColorLine / ILI9488_DrawHColorLine
        ↓ （通过 FSMC 总线，每行 600 个像素 × 2 字节 = 1200 字节，共 400 行）
Step 4: LCD 上一次性显示出完整的波形画面
```

**目的**：消除波形刷新时的闪烁（避免"先清屏再画线"导致的闪白现象）。

#### 5.2.3 不同 LCD 控制器的缓冲差异

| LCD 控制器 | 内部 VRAM | MCU 写入方式 | 效果 |
|------------|-----------|-------------|------|
| **RA8875** | 有（768KB） | 写入 RA8875 内部显存，RA8875 自动刷新屏幕 | 写入到 RA8875 的 VRAM 就像写入 Framebuffer |
| **ILI9488** | 无 | 直接驱动 LCD 面板，没有中间缓冲 | 写入即直接显示，无缓冲 |

RA8875 虽然有内部 VRAM，但这**不是 MCU 侧控制的 Framebuffer**。MCU 无法直接访问这块 VRAM 的全部内容（只能通过 8080 接口逐像素/逐行读写），所以不能像真正的 Framebuffer 那样用 DMA 一次性传输整帧。

#### 5.2.4 数据传输路径总结

```
波形数据（ADC → g_usWaveBuf）
    ↓
坐标转换（→ aPointsRe[600]）
    ↓
GUI_MEMDEV_Draw 创建 Memory Device（外部SRAM，~469KB）
    ↓
_Draw() 回调在 Memory Device 上绘制：
    ├─ DSO_DrawBakFrame() → 绘制网格（写入 Memory Device）
    ├─ GUI_DrawPolyLine() → 绘制波形（写入 Memory Device）
    └─ DSO_DrawCursorH/V() → 绘制游标（写入 Memory Device）
    ↓
emWin 自动将 Memory Device 拷贝到 LCD：
    ├─ 逐行调用 _DrawBitLine16BPP()
    ├─ → RA8875_DrawHColorLine() / ILI9488_DrawHColorLine()
    └─ 通过 FSMC 总线写入 LCD 控制器
    ↓
LCD 显示完整波形画面

非波形区域（按钮、状态栏、文字等）：
    → 直接通过 GUI_xxx 函数 → GUIDRV_Template → LCD 控制器（无缓冲）
```

### 5.3 如果要开启全屏 Framebuffer，需要做什么？

如果要实现"在 MCU 内存中渲染完整一帧（800x480x16bit = 768KB），然后一次性传输到 LCD"的方案，需要：

1. 在外部 SRAM 中预留 768KB 作为 Framebuffer（目前 2MB SRAM 足够）
2. 在 `LCDConf.c` 中定义 `#define VRAM_ADDR 0x68000000` 并设置 VRAM 大小
3. 实现 `LCD_X_SETVRAMADDR`，将 VRAM 地址配置到 LCD 控制器（仅 RA8875 支持）
4. 实现 `LCD_X_SHOWBUFFER`，支持缓冲区切换
5. 对于 ILI9488，需要自行实现 DMA 传输逻辑（ILI9488 不支持直接内存映射）

不过本项目的当前方案（波形区 Memory Device + 直接写 LCD）已经足够满足示波器的刷新需求，无需全屏 Framebuffer。

## 六、LCD 底层绘制函数的传输效率分析

### 6.1 核心机制：8080 接口的地址自增模式

RA8875 和 ILI9488 都支持 8080 并口协议。关键优化在于**进入内存写入模式后，LCD 控制器内部地址指针会自动递增**，不需要每个像素都重新设置坐标。

### 6.2 各操作的 FSMC 传输次数分析

#### 6.2.1 画单个像素（`RA8875_PutPixelGUI` / `ILI9488_PutPixel`）

**RA8875** — 5 次 FSMC 写入/像素：
```c
// LCD_RA8875.c 第 2027~2044 行
RA8875_REG = 0x46;  RA8875_RAM = _usX;       // 写入 X 坐标低8位
RA8875_REG = 0x47;  RA8875_RAM = _usX >> 8;  // 写入 X 坐标高8位
RA8875_REG = 0x48;  RA8875_RAM = _usY;       // 写入 Y 坐标低8位
RA8875_REG = 0x49;  RA8875_RAM = _usY >> 8;  // 写入 Y 坐标高8位
RA8875_REG = 0x02;  RA8875_RAM = _usColor;   // 进入写模式 + 写像素数据
```

**ILI9488** — 约 10 次 FSMC 写入/像素（SetCursor 需要 8 次 + 写命令/数据各 1 次）。

#### 6.2.2 画水平线（`RA8875_DrawHLine` / `ILI9488_DrawHLine`）

**RA8875** — 设光标 5 次 + 每像素 **1 次**（自增模式）：
```c
// LCD_RA8875.c 第 2116~2139 行
RA8875_REG = 0x46;  RA8875_RAM = _usX1;       // 设置起始X
RA8875_REG = 0x47;  RA8875_RAM = _usX1 >> 8;
RA8875_REG = 0x48;  RA8875_RAM = _usY1;       // 设置起始Y
RA8875_REG = 0x49;  RA8875_RAM = _usY1 >> 8;
RA8875_REG = 0x02;                           // 进入内存写入模式
for (i = 0; i < _usX2 - _usX1 + 1; i++) {
    RA8875_RAM = _usColor;                   // 光标自动右移，1次FSMC写 = 1像素
}
```

**ILI9488** — 设窗口 8 次 + 每像素 **1 次**（窗口内自增）：
```c
// LCD_ILI9488.c 第 576~589 行
ILI9488_SetDispWin(_usX1, _usY1, 1, _usX2 - _usX1 + 1);  // 设置1行窗口
ILI9488_REG = 0x2C;                                         // 进入写显存模式
for (i = 0; i < _usX2-_usX1 + 1; i++) {
    ILI9488_RAM = _usColor;                                 // 窗口内自动递增
}
```

#### 6.2.3 画彩色水平线（`RA8875_DrawHColorLine` / `ILI9488_DrawHColorLine`）

与水平线相同机制，只是每个像素颜色不同。这是 **emWin Memory Device → LCD 的主要传输路径**。

每行 600 个像素 = 设置光标 5 次 + 600 次数据写入。600x400 区域 = 400 行 × 605 次 = **242,000 次 FSMC 写入**。

#### 6.2.4 画垂直线（`RA8875_DrawVLine` / `ILI9488_DrawVLine`）

**效率极低**，垂直方向没有自增模式：

**RA8875** — 每像素 **9 次** FSMC 写入：
```c
// LCD_RA8875.c 第 2189~2220 行
for (i = 1; i <= _usY2 - _usY1 + 1; i++) {
    RA8875_REG = 0x02;              // 1次：写模式
    RA8875_RAM = _usColor;          // 1次：写像素
    RA8875_REG = 0x46;              // 1次：X低8位
    RA8875_RAM = _usX1;             // 1次：X值
    RA8875_REG = 0x47;              // 1次：X高8位
    RA8875_RAM = _usX1 >> 8;
    RA8875_REG = 0x48;              // 1次：Y低8位
    RA8875_RAM = _usY1 + i;
    RA8875_REG = 0x49;              // 1次：Y高8位
    RA8875_RAM = (_usY1 + i) >> 8;
}
```

**ILI9488** — 更差，每像素调用 `PutPixel`（约 10 次 FSMC 写入）。

#### 6.2.5 填充矩形

**RA8875** — **BTE 硬件加速**，约 15 条寄存器命令完成任意大小矩形填充：
```c
// LCD_RA8875.c 第 2092~2103 行
BTE_SetTarBlock(x0, y0, height, width, 0);  // 设置目标区域（8条寄存器）
BTE_SetOperateCode(0x0C);                    // 操作码：单色填充（1条寄存器）
RA8875_SetFrontColor(_usColor);              // 前景色（3条寄存器）
BTE_Start();                                 // 启动BTE引擎（2条寄存器）
BTE_Wait();                                  // 等待硬件完成
```
**无论矩形多大，FSMC 写入次数固定为约 15 次。RA8875 的 BTE 引擎在芯片内部完成所有像素填充。**

**ILI9488** — 没有硬件加速，使用窗口+连续写模式，1 次 FSMC 写入/像素：
```c
// LCD_ILI9488.c 第 711~731 行
ILI9488_SetDispWin(_usX, _usY, _usHeight, _usWidth);  // 设置窗口（8次）
ILI9488_REG = 0x2C;                                    // 写模式
for (i = 0; i < _usHeight * _usWidth; i++) {
    ILI9488_RAM = _usColor;                            // 窗口内自动递增
}
```

#### 6.2.6 清屏

**RA8875** — 使用 BTE 硬件清屏（与矩形填充相同），约 15 次 FSMC 写入清空整个屏幕。

**ILI9488** — 循环写每个像素，做了 8 倍展开优化：
```c
// LCD_ILI9488.c 第 399~430 行
n = (g_LcdHeight * g_LcdWidth) / 8;
for (i = 0; i < n; i++) {
    ILI9488_RAM = _usColor; ILI9488_RAM = _usColor;
    ILI9488_RAM = _usColor; ILI9488_RAM = _usColor;
    ILI9488_RAM = _usColor; ILI9488_RAM = _usColor;
    ILI9488_RAM = _usColor; ILI9488_RAM = _usColor;
}
```
800x480 = 384,000 像素，384,000 次 FSMC 写入。

### 6.3 效率对比总结

| 操作 | RA8875 | ILI9488 | 说明 |
|------|--------|---------|------|
| 单像素 | 5 次 | ~10 次 | SetCursor + WriteData |
| 水平线 (N像素) | 5 + N 次 | 8 + N 次 | 地址自增，**1次FSMC写/像素** |
| 垂直线 (N像素) | 9 × N 次 | ~10 × N 次 | 每像素重设坐标，**效率极低** |
| 填充矩形 (W×H) | **~15 次（固定）** | 8 + W×H 次 | RA8875 用 BTE 硬件加速 |
| 清屏 | **~15 次** | ~384,000 次 | RA8875 BTE vs ILI9488 逐像素 |
| 彩色水平线 (N像素) | 5 + N 次 | 8 + N 次 | Memory Device → LCD 的传输路径 |
| **波形区传输 (600×400)** | **400×605 ≈ 242,000 次** | **400×608 ≈ 243,200 次** | 逐行拷贝 |

### 6.4 关键结论

1. **水平方向很高效**：LCD 控制器的地址自增机制使得水平连续写只需 1 次 FSMC 写入/像素
2. **垂直方向非常低效**：没有垂直自增，每个像素需要重新设置坐标（5~10次 FSMC 写入/像素）
3. **RA8875 的 BTE 引擎很强**：矩形填充、清屏等操作几乎不需要 CPU 参与
4. **没有使用 DMA**：所有传输都是 CPU 通过 FSMC 内存映射直接写，没有 DMA 批量搬运
5. **波形区 Memory Device 传输**：emWin 按行拷贝（`_DrawBitLine16BPP`），600x400 区域约 24 万次 FSMC 写入，在 168MHz 主频下约需几毫秒完成

## 七、FSMC + 外置 SRAM + emWin 运行稳定性分析与 SRAM 选型对比

### 7.1 问题现象

使用 STM32F407 + FSMC + 外置 SRAM 运行 emWin 时，程序运行一段时间后随机进入 BusFault 死机。这是一个非常常见的嵌入式稳定性问题。

### 7.2 BusFault 的六大常见原因（按可能性排序）

#### 原因一：FSMC 时序余量不足（最可能，约占 80%）

本项目 FSMC 时序配置（`bsp_fsmc_sram.c`）：

```c
p.FSMC_AddressSetupTime = 3;   // 3 HCLK
p.FSMC_DataSetupTime = 2;      // 2 HCLK  ← 非常紧张
p.FSMC_BusTurnAroundDuration = 1;
```

168MHz 主频下，1 HCLK ≈ 5.95ns，2 HCLK ≈ **11.9ns**。

本项目用的 SRAM 是 IS61WV102416BLL-**10**TL，后缀 10 表示访问时间 10ns。11.9ns 的数据建立时间相比 10ns 的 SRAM 速度，**余量不到 2ns**。在温度升高、电压波动、PCB 寄生参数等因素影响下，极易出现偶发读写错误。

**解决建议**：
```c
p.FSMC_AddressSetupTime = 4;   // 加大地址建立时间
p.FSMC_DataSetupTime = 6;      // 至少 6，保守给 8（约 48ns，远大于 SRAM 的 10ns）
p.FSMC_BusTurnAroundDuration = 2;
```
放宽时序后牺牲的带宽微乎其微（168MHz 下每个周期 6ns，从 4 周期增加到 10 周期只慢了约 36ns），但稳定性大幅提升。

#### 原因二：硬件信号完整性问题

FSMC 是高速 16 位并行总线（16 根数据线 + 多根地址线 + 控制线），在 PCB 上走线长且密集时容易出现：
- **信号反射/串扰**：长走线 + 不匹配的阻抗导致信号过冲/下冲
- **容性负载过重**：FSMC 总线同时驱动 LCD 控制器 + SRAM，负载加倍
- **浮空信号**：未使用的地址线/数据线浮空产生毛刺

**排查建议**：
- 数据线和地址线串联 22Ω~33Ω 电阻做源端匹配
- NE（片选）信号加 10KΩ 上拉
- 检查 PCB 走线等长、阻抗控制
- 用逻辑分析仪抓 FSMC 时序，确认建立/保持时间满足要求

#### 原因三：emWin 栈溢出 / 内存越界

emWin 的 Memory Device、字体缓存、窗口管理器等会在运行时动态分配内存。如果这些内存分配在外置 SRAM 中且超出了实际 SRAM 容量，访问不存在的地址就会 BusFault。

**排查建议**：
- 确认外置 SRAM 的实际容量与代码中定义的大小一致
- 在 emWin 初始化时调用 `GUI_ALLOC_GetNumFreeBytes()` 监控剩余内存
- 检查 `GUI_CONF_H` 中 `GUI_NUMBYTES` 和 `GUI_MAX_TASKS` 的配置是否合理

#### 原因四：uCOS-III 任务栈被损坏

如果 uCOS-III 的任务栈放在外置 SRAM 中，一旦 SRAM 读写偶发出错，栈中的返回地址、寄存器值被破坏，任务切换时就会跑到非法地址 → BusFault。

**本项目的安全做法**：
- 链接器中**没有把外置 SRAM 作为通用 RAM**（`DATA_IN_ExtSRAM` 被注释掉）
- `RW_IRAM1` 只有内部 128KB（`0x20000000`），外置 SRAM 仅用于专用缓冲
- 栈和堆全部在内置 SRAM 中，不会被 SRAM 读写错误影响

**建议**：确保你的项目也**不要把栈/堆放到外置 SRAM**。

#### 原因五：DMA 与 CPU 总线仲裁冲突

如果同时有 DMA 操作也在访问 FSMC 总线（如 DMA 搬运 ADC 数据到 SRAM），DMA 和 CPU 同时抢 AHB 总线可能导致时序 violation。

**排查建议**：用互斥量或时间片错开 DMA 和 CPU 对 FSMC 的访问。

#### 原因六：供电不稳定

外置 SRAM + LCD 控制器的瞬间功耗可达数百 mA，3.3V 电源纹波过大会导致 SRAM 工作异常。

**排查建议**：用示波器测量 3.3V 电源纹波，确保在 50mV 以内。

### 7.3 SRAM 型号对比

| 参数 | **XM8A51216V33A**（你的） | **IS61WV102416BLL-10TL**（本项目的） | **IS62WV51216**（参考对比） |
|------|--------------------------|--------------------------------------|---------------------------|
| **容量** | 512K × 16bit = **1MB** | 1024K × 16bit = **2MB** | 512K × 16bit = **1MB** |
| **访问速度** | **10ns / 12ns** | **10ns** | 45ns / 55ns |
| **数据宽度** | 16 bit | 16 bit | 16 bit |
| **电压** | 3.3V | 3.3V | 3.3V |
| **封装** | TSOP2(44) / TSOP1(48) | TSOP | - |
| **接口** | 异步 SRAM | 异步 SRAM | 异步 SRAM |

#### 型号命名规则

```
XM8A51216V33A  →  512K (words) × 16bit = 8Mbit = 1MB, 3.3V, 10/12ns
IS61WV102416   →  1024K (words) × 16bit = 16Mbit = 2MB, 10ns
IS62WV51216    →  512K (words) × 16bit = 8Mbit = 1MB, 45/55ns
         ^^^^^ ^^^
         容量  位宽
```

#### 速度对比（168MHz 下 FSMC 时序需求）

168MHz 主频，1 HCLK ≈ 5.95ns：

| SRAM | 访问时间 | FSMC DataSetupTime 最小值 | 本项目实际设置 | 余量 |
|------|---------|--------------------------|---------------|------|
| **XM8A51216V33A (10ns)** | 10ns | 10/5.95 ≈ **2 个 HCLK** | 2 个 HCLK | **≈ 0ns（无余量）** |
| **IS61WV102416BLL-10 (10ns)** | 10ns | ≈ **2 个 HCLK** | 2 个 HCLK | **≈ 0ns（无余量）** |
| IS62WV51216 (45ns) | 45ns | 45/5.95 ≈ **8 个 HCLK** | — | 需要大幅放宽时序 |

> **关键发现**：XM8A51216V33A 的 10ns 速度等级与本项目 IS61WV102416BLL-10TL **完全相同**。本项目用 `DataSetupTime = 2` 跑起来没问题，说明你的芯片在速度上不是瓶颈。但余量接近零，温度/电压波动时仍可能不稳定。

#### 关键风险点

1. **容量只有 1MB（项目的 2MB 的一半）**：本项目 2MB SRAM 上需要存放波形区 Memory Device（~469KB）、ADC 采样缓冲、emWin GUI 内存池等。如果你的 1MB SRAM 不够用，emWin 分配的显存可能越界到 `0x68100000` 以上的不存在的地址，**直接触发 BusFault**。

2. **时序余量几乎为零**：10ns SRAM + 168MHz + `DataSetupTime = 2`（≈12ns），只有 ~2ns 余量。本项目能跑是因为它用的是 2MB 大品牌 SRAM，品控好。即使你的芯片也是 10ns，建议适当放宽到 `DataSetupTime = 3~4`，留一点安全裕度。

3. **品牌与品控**：IS61WV 系列是 ISSI 大规模量产产品，一致性有保障。XM8 系列需确认来源正规、非翻新片。

### 7.4 排查步骤（推荐顺序）

| 步骤 | 操作 | 目的 |
|------|------|------|
| **1** | 确认 `EXT_SRAM_SIZE` 定义为 `1*1024*1024`（不超过 1MB） | **排除内存越界** |
| **2** | 放宽 FSMC 时序（DataSetupTime = 6~8） | 排除时序余量不足 |
| **3** | 在 `HardFault_Handler` 中读取 `SCB->BFAR` | 确认 BusFault 发生的地址 |
| **4** | 用 `GUI_ALLOC_GetNumFreeBytes()` 监控 emWin 内存使用 | 确认 1MB 不够用 |
| **5** | 确认栈/堆不在外置 SRAM 中 | 排除栈损坏 |
| **6** | 运行 SRAM 读写校验（类似 `bsp_TestExtSRAM()`） | 验证 SRAM 硬件可靠性 |
| **7** | 检查 FSMC 信号完整性（逻辑分析仪） | 排除硬件问题 |
| **8** | 检查 3.3V 电源纹波 | 排除供电问题 |

#### HardFault 调试代码

```c
void HardFault_Handler(void) {
    volatile uint32_t bfar = SCB->BFAR;   // BusFault 地址
    volatile uint32_t cfsr = SCB->CFSR;   // Configurable Fault Status
    volatile uint32_t hfsr = SCB->HFSR;   // HardFault Status
    volatile uint32_t mmar = SCB->MMFAR;  // MemManage Fault Address
    volatile uint32_t sp   = __get_MSP(); // 当前栈指针

    // 用串口打印这些值，或设置断点观察
    // BFAR 在 0x68000000~0x680FFFFF 范围 → 外置 SRAM 问题（1MB = 0x68000000~0x680FFFFF）
    // BFAR 超出 0x680FFFFF → 内存越界（1MB SRAM 被当成 2MB 用了）
    // BFAR 在其他范围 → 可能是栈损坏或其他问题
    while(1);
}
```

如果 `BFAR` 读出为 `0x68100000` 或更大，说明代码访问了超过 1MB 的范围——**确认你的 SRAM 实际只有 1MB，这就是 BusFault 的直接原因**。

### 7.5 BusFault 根因确认：V2 开发板 + V3 例程的时序不匹配

#### 问题本质

| | V2 开发板（你的硬件） | V3 开发板（例程对应的硬件） |
|--|----------------------|---------------------------|
| **SRAM 芯片** | **IS62WV51216**（45ns/55ns） | **XM8A51216V33A**（10ns/12ns） |
| **访问速度** | 慢 4~5 倍 | 快 |
| **例程 FSMC 时序** | —（例程不是给 V2 写的） | `DataSetupTime = 2`（≈12ns） |

**V3 例程的 FSMC 时序是为 10ns 的 XM8A51216V33A 优化的，`DataSetupTime = 2`（≈12ns）。但你 V2 板子上的 IS62WV51216 最少需要 45ns，实际只给了 12ns——严重欠时序。**

#### 为什么不是立即崩溃而是"跑久了才死"

- 低频操作（初始化、少量 UI 刷新）时，SRAM 偶尔能"碰巧"在 12ns 内返回数据（SRAM 内部实际响应可能比标称 45ns 快，只是不保证）
- 高频使用 LCD（emWin 大面积刷新、Memory Device 拷贝）时，FSMC 连续快速访问 SRAM，时序违规的概率大幅增加
- 数据一旦出错，可能破坏 emWin 的内存管理结构或指针，程序继续运行一段时间后才崩溃到 BusFault

#### 解决方案

将 FSMC 时序改为适配 IS62WV51216（45ns）的配置：

```c
// 168MHz 下，1 HCLK ≈ 5.95ns
// IS62WV51216 需要 45ns → 45 / 5.95 ≈ 7.6 → 至少 8 个 HCLK

p.FSMC_AddressSetupTime = 5;      // 地址建立时间：5 HCLK（≈30ns）
p.FSMC_DataSetupTime = 8;         // 数据建立时间：8 HCLK（≈48ns）← 关键！从 2 改为 8
p.FSMC_BusTurnAroundDuration = 3; // 总线周转时间：3 HCLK（≈18ns）
```

> **注意**：时序放宽后 FSMC 带宽会下降。单次访问从 ~4 个 HCLK 增加到 ~16 个 HCLK，带宽降为原来的 1/4。对于 emWin 的 Memory Device 传输（约 24 万次 FSMC 写入），刷新时间会从几毫秒增加到十几毫秒。如果示波器波形刷新率要求不高（30fps 以下），完全可以接受。

#### 其他需要检查的地方

如果 V2 例程和 V3 例程之间还有其他硬件差异（如 LCD 控制器型号、FSMC Bank 选择等），也需要同步修改。

## 八、FSMC 与异步 SRAM 的交互原理

### 8.1 什么是 FSMC

FSMC（Flexible Static Memory Controller，灵活的静态存储器控制器）是 STM32F407 内部的一个外设，它的作用是**把 MCU 内部的存储器访问操作（读写某个地址）翻译成外部总线的信号时序**，让 CPU 可以像访问内部 RAM 一样访问外部 SRAM。

```
CPU 执行:  *ptr = 0x1234;    （ptr 指向 0x68000000，即外部 SRAM）
                    ↓
           CPU 发出 AHB 总线读写请求
                    ↓
           FSMC 自动产生以下信号序列：
           ┌─────────────────────────────────────────┐
           │  NE3（片选）= 0    ← 选中 Bank3 的芯片    │
           │  A[0:24]（地址）= 目标地址               │
           │  D[0:15]（数据）= 0x1234                │
           │  NOE（读使能） 或 NWE（写使能）= 0        │
           │  NBL0/NBL1（字节使能）= 0               │
           └─────────────────────────────────────────┘
                    ↓
           外部 SRAM 芯片收到信号，完成数据读写
                    ↓
           FSMC 读回数据（读操作时），返回给 CPU
```

对 CPU 来说，`*(uint16_t *)0x68000000 = 0x1234` 就是一条普通的内存写入指令，FSMC 在背后自动处理所有总线时序。

### 8.2 FSMC 与 SRAM 的硬件连接

本项目使用 **FSMC Bank1-NORSRAM3**（片选 NE3/PG10），连接关系：

```
STM32F407                                    IS61WV102416BLL / XM8A51216V33A / IS62WV51216
┌──────────┐                                  ┌──────────────────┐
│          │  D[0:15]  (16根数据线)            │ D0~D15           │
│          ├─────────────────────────────────→│                  │
│          │  A[0:18]  (19根地址线)            │ A0~A18           │
│          ├─────────────────────────────────→│                  │
│  FSMC    │  NE3 (PG10)  片选信号             │ CE# (芯片使能)    │
│          ├─────────────────────────────────→│                  │
│          │  NOE        读使能                │ OE# (输出使能)    │
│          ├─────────────────────────────────→│                  │
│          │  NWE        写使能                │ WE# (写使能)      │
│          ├─────────────────────────────────→│                  │
│          │  NBL0/NBL1  字节使能              │ LB#/UB#          │
│          ├─────────────────────────────────→│                  │
└──────────┘                                  └──────────────────┘

所有信号低电平有效（后缀 # 表示 active low）
```

- **16 根数据线**：一次传输 16bit（2 字节）
- **19 根地址线**：可寻址 2^19 = 512K 个 16bit 单元 = 1MB（本项目的 2MB 芯片用了 20 根地址线）
- **NE3**：FSMC Bank3 的片选，当 CPU 访问 `0x68000000` 起始的地址空间时，NE3 自动拉低

### 8.3 异步读时序（Mode A）

当 CPU 执行 `uint16_t val = *ptr;` 读取外部 SRAM 时，FSMC 按以下时序发送信号：

```
CLK (168MHz)  ─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─
               │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │

Address        ─────┐                         ┌─────
  (A[0:18])         ├─ ADDR ──────────────────┤
                     │                         │

NE# (片选)      ─────┐                         ┌─────
                    ├───────────────────────────┤
                    │                          │

NOE# (读使能)    ─────────┐                  ┌─────
                       ├────── READ ──────────┤
                       │                      │

Data (D[0:15])  ────────────────┐          ┌───
  (SRAM输出)                     ├── DATA ─┤
                                 │          │
                                 │← tAVAV →│← tDVR →│
```

关键时序参数：

| FSMC 参数 | 对应信号 | 含义 |
|-----------|---------|------|
| `AddressSetupTime` | tAVAV | 地址有效到 NOE# 有效的时间（地址建立时间） |
| `DataSetupTime` | tDVR | NOE# 有效到数据有效的时间（数据建立时间，**SRAM 必须在这个时间内输出数据**） |
| `BusTurnAroundDuration` | | 两次连续访问之间的间隔 |

### 8.4 异步写时序（Mode A）

当 CPU 执行 `*ptr = 0x1234;` 写入外部 SRAM 时：

```
CLK (168MHz)  ─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─┬─
               │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │

Address        ─────┐                         ┌─────
                     ├─ ADDR ──────────────────┤

NE# (片选)      ─────┐                         ┌─────
                    ├───────────────────────────┤

NWE# (写使能)    ─────────┐                  ┌─────
                       ├────── WRITE ─────────┤
                       │                      │

Data (D[0:15])  ───────────┐               ┌───
  (MCU输出)                  ├── 0x1234 ───┤
                              │              │
                              │← tAVAV →│← tD VW →│
```

写时序的 `DataSetupTime` 含义与读不同：它是指 **NWE# 有效的脉冲宽度**（MCU 驱动数据到总线上的时间），SRAM 必须在 NWE# 上升沿采样数据。

### 8.5 时序计算：为什么 V2 板子会 BusFault

以本项目 V3 的配置为例（168MHz，1 HCLK ≈ 5.95ns）：

#### V3 配置（适配 XM8A51216V33A，10ns 芯片）

```
AddressSetupTime = 3  →  3 × 5.95ns = 17.85ns（地址建立）
DataSetupTime    = 2  →  2 × 5.95ns = 11.90ns（数据建立）
BusTurnAround    = 1  →  1 × 5.95ns =  5.95ns（总线周转）
                              总计 ≈ 35.75ns / 次访问
```

**SRAM 需要 10ns，给了 11.9ns → 余量 1.9ns（勉强能工作）**

#### 同样的配置用在 V2 板子（IS62WV51216，45ns 芯片）

```
AddressSetupTime = 3  →  17.85ns
DataSetupTime    = 2  →  11.90ns  ← SRAM 需要 45ns！！！
BusTurnAround    = 1  →  5.95ns
```

**SRAM 需要 45ns，只给了 11.9ns → 严重不足！差额 33ns**

#### V2 应该的配置

```
AddressSetupTime = 5  →  5 × 5.95ns = 29.75ns（≥ 地址建立需求）
DataSetupTime    = 8  →  8 × 5.95ns = 47.60ns（≥ 45ns，余量 2.6ns）
BusTurnAround    = 3  →  3 × 5.95ns = 17.85ns
                              总计 ≈ 95.25ns / 次访问
```

### 8.6 FSMC 各时序参数详解

```
读时序（Mode A）：

         ← ADDSET →← DATAST →
         ┌─────────┬──────────┬────────┐
  CLK    │         │          │        │
  ADDR   ├─────────┤──────────┤        │
  NOE#              ├──────────┤        │
  DATA                         ├────────┤
                               ↑        ↑
                          SRAM输出   MCU采样
                          数据     数据

写时序（Mode A）：

         ← ADDSET →← DATAST →
         ┌─────────┬──────────┬────────┐
  CLK    │         │          │        │
  ADDR   ├─────────┤──────────┤        │
  NWE#              ├──────────┤        │
  DATA              ├──────────┤        │
                    ↑          ↑
               MCU输出     MCU释放
               数据       数据
```

| 参数 | 寄存器位 | 最小值 | 含义 |
|------|---------|--------|------|
| **AddressSetupTime (ADDSET)** | BTR[3:0] | 1 | 地址线有效后，到 NOE#/NWE# 拉低前的等待周期。SRAM 需要时间锁存地址 |
| **AddressHoldTime (ADDHLD)** | BTR[7:4] | 0 | Mode A 下不用（设为 0）。Mode B/C/D 才有意义 |
| **DataSetupTime (DATAST)** | BTR[15:8] | 2（实际最小 1） | **最关键的参数**。读：从 NOE# 有效到 MCU 采样数据的时间。写：NWE# 的有效脉冲宽度 |
| **BusTurnAroundDuration (BUSTURN)** | BTR[19:16] | 0 | 读写方向切换时的额外等待周期。连续读→写或写→读时需要 |
| **AccessMode** | BTR[29:28] | A | 时序模式。Mode A 最简单：地址、数据不复用 |
| **CLKDivision** | BTR[23:20] | 0 | Mode A 下不用（同步模式才需要） |
| **DataLatency** | BTR[27:24] | 0 | Mode A 下不用（同步模式才需要） |

### 8.7 连续写操作的实际过程（以 emWin 刷 LCD 为例）

当 emWin 通过 Memory Device 把波形数据拷贝到 LCD 时，实际上是在连续写外部 SRAM（LCD 控制器也挂在 FSMC 上）。FSMC 的工作过程：

```c
// CPU 执行：*(volatile uint16_t *)0x68000000 = 0xF800;
// 等价于 FSMC 完成以下操作：

// 第1步：地址建立（ADDSET = 3 个 HCLK）
//   A[0:18] = 0x00000    ← 地址线输出目标地址
//   NE3#    = 0           ← 片选拉低，选中 SRAM
//   （等待 3 个时钟周期，让地址稳定）

// 第2步：数据写入（DATAST = 2 个 HCLK）
//   D[0:15] = 0xF800     ← 数据线输出要写入的数据
//   NWE#    = 0           ← 写使能拉低
//   （等待 2 个时钟周期，让 SRAM 完成写入）
//   NWE#    = 1           ← 写使能恢复高电平（SRAM 在上升沿锁存数据）

// 第3步：总线周转（BUSTURN = 1 个 HCLK）
//   （等待 1 个时钟周期，让总线信号稳定后再进行下一次访问）

// 总耗时：3 + 2 + 1 = 6 个 HCLK ≈ 35.7ns
```

连续写 600 个像素（一行波形数据）时，FSMC 重复上述过程 600 次：

```
时间 →  0    6H   12H   18H   24H  ...  3594H
       ├─────┼─────┼─────┼─────┼─────┼─────┤
       写P0  写P1  写P2  写P3  写P4  ...  写P599

总时间 = 600 × 6 HCLK = 3600 HCLK ≈ 21.4μs（V3 配置）
总时间 = 600 × 16 HCLK = 9600 HCLK ≈ 57.1μs（V2 修正配置）
```

### 8.8 "跑久了才死"的原因

FSMC 时序不满足时，并不是每次访问都出错，而是**概率性问题**：

```
SRAM 实际响应时间分布（以 IS62WV51216 为例）：

  概率
  100% ┤
       │                    ┌───── 45ns 以内完成（datasheet 保证）
   50% ┤               ┌────┤
       │          ┌────┤    └───── 30~40ns 完成（典型值，大部分情况）
       │     ┌────┤
    0% ┤─────┤                      ← 10~20ns 完成（极端情况，极少）
       └─────┴─────┴─────┴─────┴─────┴─────
            0    10    20    30    40    50  ns
                      ↑
               FSMC 只等了 12ns
               大部分时候已经出错了！
               但偶尔能碰巧读到正确数据
```

- **低频操作**：两次访问之间间隔长，SRAM 有充足时间恢复，出错概率低
- **高频连续访问**：FSMC 几乎不间隔地连续发读写请求，时序违规的概率累积，出错概率急剧升高
- **出错后果**：读到的数据是垃圾值 → 如果是 emWin 的内存管理结构被破坏 → 指针跑飞 → BusFault
